As 3 ideias que sustentam tudo

  • O relacionamento é cidadão de primeira classe. No grafo ele é armazenado como aresta; no relacional ele é recalculado com JOIN a cada consulta.
  • Index-free adjacency. Cada nó aponta direto pros vizinhos → seguir uma aresta é O(1) local, independente do tamanho do banco. Por isso o multi-hop não degrada.
  • A tradução é trivial; o desempenho, não. SQL→Cypher é fácil de escrever, mas o tempo diverge de empate (busca rasa) a centenas de vezes (multi-hop), até onde o SQL não termina ou não consegue expressar.

04Relacional vs. Grafo~1min30
relacional · 3 tabelas + JOIN
SELECT e.nome FROM pessoas p
 JOIN trabalha_em t ON t.pessoa_id=p.id
 JOIN empresas e ON e.id=t.empresa_id
WHERE p.nome='Ana';
grafo · o padrão é o desenho
MATCH (p:Pessoa {nome:'Ana'})
      -[:TRABALHA_EM]->(e:Empresa)
RETURN e.nome;
fale
"Mesma informação, dois modelos. O ponto que importa: cada 'pulo' a mais — amigo do amigo do amigo — no relacional é mais um JOIN; no grafo é só estender o desenho. É daí que vem a diferença de desempenho."
05"Olá, mundo" em Cypher~1min30
no slide
// 1 · criar
CREATE (ana:Pessoa {nome:'Ana'}),
       (acme:Empresa {nome:'ACME'}),
       (ana)-[:TRABALHA_EM {desde:2020}]->(acme);
// 2 · consultar
MATCH (p:Pessoa)-[:TRABALHA_EM]->(e:Empresa) RETURN p.nome, e.nome;
fale
"Cypher é a 'SQL dos grafos'. A sintaxe é ASCII-art: parênteses são nós, setas são relacionamentos. CREATE grava; MATCH...RETURN casa o padrão e devolve. Você descreve o desenho que procura — o banco acha."

Opcional ao vivo (Neo4j Browser, localhost:7474): cole o CREATE, depois o MATCH, mostre o nó e a aresta desenhados.

09Multi-hop — o pulo do gato~2min · ao vivo
a pergunta

1 hop: empresas que a pessoa controla. 2 hops com volta: "quem mais usou o mesmo intermediário?" — o padrão investigativo clássico.

// Cypher: um padrão
MATCH (o:Officer)-[:officer_of]->(e1)<-[:intermediary_of]-(i)
      -[:intermediary_of]->(e2)<-[:officer_of]-(outro:Officer)
RETURN DISTINCT outro.name;
// O mesmo em SQL: 4 JOINs sobre a tabela de relações
fale
"Mesma pergunta. No Cypher é um padrão; o mesmo em SQL já são 4 JOINs. Um lado é um desenho, o outro é um quebra-cabeça de junções."
10Caminho mais curto entre 2 pessoas~1min30 · ao vivo
MATCH p = shortestPath( (a)-[*..6]-(b) ) RETURN p;   // Cypher: 1 linha
// SQL: WITH RECURSIVE … CTE que reexpande a fronteira a cada nível
fale
"O caso onde o SQL sofre: a profundidade é desconhecida. No Cypher é shortestPath, uma linha. No SQL é um CTE recursivo cuja fronteira explode. A partir de um nó bem conectado, o SQL estoura o timeout já em 3 hops; o Cypher responde em ~6ms em qualquer profundidade."

Conceitos & contexto (pra entender, não decorar)

🔤 O que é Cypher, exatamente

A linguagem de consulta do Neo4j — o que o SQL é pro relacional. Três marcas:

  • Declarativa — você diz o quê quer, não o como buscar (igual SQL nesse ponto).
  • ASCII-art — você desenha o padrão: ( ) é um nó, -[ ]-> é uma relação com direção. O código parece o diagrama.
  • Padrão ISO — desde 2024 virou o GQL (Graph Query Language), aprovado pela ISO — mesmo comitê do SQL. Versão aberta: openCypher.

🧠 Index-free adjacency — o conceito que explica tudo

Cada nó guarda, fisicamente, ponteiros diretos pros vizinhos. Seguir uma aresta é O(1) local — ter 2 milhões ou 2 bilhões de nós não muda o custo do pulo. Por isso o multi-hop não degrada. No SQL, cada hop é um JOIN que varre/indexa a tabela de relações inteira de novo.

Analogia da agenda. Grafo = o telefone dos amigos salvo no seu celular (instantâneo). JOIN do SQL = toda vez, procurar o nome numa lista telefônica da cidade inteira. Com 1 amigo tanto faz; com "amigo do amigo do amigo", a lista telefônica te mata.

Por que o caminho variável quebra o SQL: a fronteira do CTE recursivo cresce exponencialmente → problema algorítmico, não de índice. Índice não resolve. (Decore isso.)

Perguntas prováveis (Q&A)

Por que o grafo é mais rápido no multi-hop?
Index-free adjacency: cada nó guarda ponteiros diretos pros vizinhos. Seguir uma aresta é O(1) local, independente do tamanho total do banco. No SQL, cada hop é um JOIN que varre/indexa a tabela de relações inteira de novo.
Então é sempre melhor usar grafo?
Não. Pra dado tabular, agregação e relatório, o relacional é mais simples e eficiente. Grafo ganha quando o relacionamento é o protagonista.
O Postgres não ficaria mais rápido com mais índices/tuning?
Ajuda nos hops rasos. Mas o problema do caminho de profundidade variável é algorítmico (a fronteira do CTE cresce exponencialmente) — índice não resolve. E PageRank/Louvain não têm equivalente declarativo.
O que é GDS e APOC?
GDS = Graph Data Science, a lib de algoritmos (PageRank, Louvain, caminhos). APOC = biblioteca de utilitários/procedimentos do Neo4j.
Os dados são reais? Como garantem que é a mesma comparação?
Sim — ICIJ Offshore Leaks, licença ODbL, 2M nós / 3,3M relações. O Neo4j é a fonte; um ETL (pnpm mirror:pg) espelha exatamente os mesmos nós e relações pro schema relacional. O id do Neo4j vira node_id no Postgres — mesma chave dos dois lados.
O que é Cypher?
A linguagem de consulta declarativa do Neo4j (hoje padronizada como GQL/openCypher). Descreve padrões de nós e relacionamentos em sintaxe ASCII-art.
